# V1 — Bâtir le Cahier de Textes Interactif à partir de zéro

> Blueprint complet et ordonné. Chaque phase est **livrable seule** et prépare la suivante.
> Stack : **Vite 6 · React 19 · TypeScript · Tailwind v4 (build) · Vercel (Hobby) · Upstash Redis**.

---

## Vue d'ensemble — l'architecture en une image

```
┌─────────────────────────  NAVIGATEUR (source de vérité)  ─────────────────────────┐
│                                                                                    │
│  localStorage                    React (hash routing #/ et #/classe/{id})          │
│  ├─ classManager_v1  ◄────────►  Dashboard (cartes de classes par cycle)           │
│  ├─ classData_v1_{id} ◄───────►  Editor (table Date|Contenu|Remarque)              │
│  ├─ appConfig_v1     ◄────────►  ConfigModal (onglets) / emploi du temps           │
│  └─ syncMeta_v1                  ▲                                                 │
│         │                        │ utils/progression.ts (UNE source de calcul)     │
│         │  syncBus (dirty)       │ utils/lateness.ts + calendar.ts (retard)        │
│         ▼                        │ utils/dateValidation.ts (garde de saisie)       │
│  SyncContext (debounce 20 s) ────┘                                                 │
└──────────┬─────────────────────────────────────────────────────────────────────────┘
           │ push/pull JSON (cookies httpOnly)
┌──────────▼──────────────────  VERCEL (5 fonctions)  ───────────────────────────────┐
│  api/auth.ts   api/sync.ts   api/admin.ts   api/notify.ts (cron 18h)   send-email  │
│                     │                                                              │
│              ┌──────▼──────── UPSTASH REDIS ────────────────┐                       │
│              │ user:{phone}            (compte)             │                       │
│              │ classes:{phone}         (classes+emploi du t)│                       │
│              │ lessons:{phone}:{id}    (1 blob par classe)  │                       │
│              │ admin:snapshots  (hash) (1 cmd pour N profs) │                       │
│              │ push:subs        (hash) (abonnements push)   │                       │
│              └──────────────────────────────────────────────┘                       │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

**Trois principes structurants** :
1. **Offline-first** — le localStorage est la vérité pendant l'édition ; le cloud est une copie + un instantané compact. L'app marche dans le train.
2. **Une seule source de calcul** — la progression (`progression.ts`), le calendrier (`calendar.ts`), le retard (`lateness.ts`) et la validation (`dateValidation.ts`) sont des **modules purs**, sans React ni DOM, importables aussi bien par les composants que par les fonctions serverless. Zéro duplication client/serveur.
3. **Le serveur lit peu** — l'admin et le cron ne lisent jamais les gros blobs : uniquement le hash `admin:snapshots` (1 commande Redis pour N enseignants).

## Phase 0 — Socle du projet

1. `npm create vite@latest` (react-ts), puis : `immer use-immer framer-motion better-react-mathjax`.
2. Tailwind v4 : `npm i -D tailwindcss @tailwindcss/vite` ; dans `index.css` → `@import "tailwindcss";` + `@theme inline { ... }` avec les tokens (primary `#C96442`, gold `#B8935A`, fonds sable). **Jamais de CDN.**
3. Icônes `lucide-react` via un **barrel** `components/ui/icons.ts` ; toasts `sonner`.
4. Alias `@/` dans `vite.config.ts` + `tsconfig.json` (`resolveJsonModule: true`).

> ⚠️ Tailwind v4 scanne les **chaînes complètes** : `bg-emerald-100` ✅, `` bg-${color} `` ❌ (utiliser des maps de chaînes).

## Phase 1 — Modèle de données & table locale

**Le cœur du logiciel.** Types dans `types.ts` :

```
ClassInfo { id, name, subject, cycle: 'college'|'lycee'|'prepa', color, createdAt }
TopLevelItem (chapter | devoir_maison | controle_continu | ...) → sections[]
Section → subsections[] | items[]        ← arbre à 4 niveaux max
LessonItem { type, number, title, description, page, date?, remark?, separatorAfter? }
Indices { chapterIndex, sectionIndex?, subsectionIndex?, subsubsectionIndex?, itemIndex? }
```

- `utils/dataUtils.ts` : `findItem(data, indices)` (traversée), `addItem/addSection/addTopLevelItem` (insertion après un ancre), `moveWithinParent` (réordonner entre frères), `flattenLessons` (arbre → liste plate).
- **Rendu** : `MainTable` aplatit l'arbre (`FlatDataItem[]` avec `Indices` par ligne) puis rend `TableRow`. Fusion visuelle des dates consécutives identiques (`applyDateMerges` : méta `isStart/isEnd/count`).
- **État** : `useHistoryState` (Immer + pile de 50 instantanés → undo/redo). Persistance `classData_v1_{id}` avec auto-save débouncé 1,5 s.
- **Virtualisation fenêtre** (`ui/virtual-list.tsx`) au-delà de 140 lignes ; option `keepIndices` pour épingler la ligne en cours d'édition.

> 💡 La liste de niveaux officiels (1AC…2BAC PC/SVT/Sc Maths, MPSI…) vit dans `constants.ts` (`CLASS_LEVELS_BY_CYCLE`) — la création de classe est un choix guidé, pas un champ libre.

## Phase 2 — Backend & authentification (téléphone + mot de passe)

1. `api/_lib/` (non déployé) : `http.ts` (types + `HttpError`), `redis.ts` (client + registre `KEYS`), `auth.ts` (**scrypt natif** + JWT `jose` + cookies httpOnly), `validate.ts` (téléphone → chiffres purs, 8-15).
2. `api/auth.ts` : actions `register` (SET NX → 409 si pris ; cycles + matières choisis à l'inscription), `login` (rate-limit INCR/EXPIRE 10/5 min, message d'erreur unique), `logout`, `me`.
3. Client : `AuthContext` (`loading/authenticated/anonymous/offline` — erreur réseau + cache → mode offline), `AuthPage` (2 onglets).
4. **Interrupteur** `config/features.ts → AUTH_REQUIRED` (voir README §6) : `false` pendant la construction, `true` = page d'auth active. Une ligne.

## Phase 3 — Synchronisation offline-first

1. `utils/progression.ts` : `computeProgressionStats` (complétion % = items datés/total, séances = dates distinctes, détail par chapitre) + `computeTeacherSnapshot` (instantané compact ~150 o + 120 o/classe).
2. `utils/syncBus.ts` : bus d'événements module-level (`markClassDirty/markClassesListDirty/markClassDeleted` + compteur de version, pas un booléen).
3. `SyncContext` : greffé sur `Editor.saveData` et `useClassManager` ; push débouncé 20 s (classes + blobs sales + snapshot), flush sur `pagehide` ; pull au démarrage avec **LWW par classe** (`syncMeta_v1` local vs `classMeta` serveur) ; événement `pull-applied` pour recharger les hooks.
4. `api/sync.ts` : POST = pipeline Redis (SET classes / SET lessons sales / DEL supprimées / HSET snapshot, téléphone imposé côté serveur) ; GET = liste légère ou `?classId=` un blob.

> 💡 Budget : ~60-90 K commandes/mois pour 30 profs actifs (quota gratuit 500 K). Garde 950 Ko/requête.

## Phase 4 — Emploi du temps, calendrier, retard

1. `public/vacances-jourferie.json` : année scolaire + fériés (`approximatif: true` pour les religieux) + vacances. Client : fetch `no-cache` (corrigeable sans rebuild) ; serveur : import statique.
2. `utils/calendar.ts` (pur) : `isSchoolDay`, `countExpectedSessions`, `nextSchoolDay`, `todayInMorocco` (⚠ serverless = UTC).
3. Emploi du temps = grille jours × créneaux (`TimetableEntry{day, slot, classId}`) dans `config.timetable`, saisie dans l'onglet dédié du ConfigModal, **dérivée** en `schedules` par `deriveSchedules()` — le moteur de retard ne connaît que `schedules`.
4. `utils/lateness.ts` : attendu (emploi du temps − vacances − fériés − **absences** via `withAbsences`) vs saisi → `ok/notice/warning/critical` + messages français → `LatenessBanner` (Dashboard) + snapshot.

## Phase 5 — Intelligence de saisie

- `utils/dateValidation.ts` : `validateSessionDate(date, classe, config)` → alertes *pas-ce-jour-là / férié / vacances / absence / hors année*.
- **Un seul validateur mémoïsé dans l'Editor**, branché sur 4 circuits : modale d'affectation (live), formulaire inline (live), cellule date et séparateur (toasts). Non bloquant.
- Impression intelligente : `printMeta.ts` mémorise les dates imprimées par classe → « imprimer seulement les nouveautés ? ».

## Phase 6 — Admin

- `admin.html` = 2ᵉ entrée Vite (`rollupOptions.input`), mini-app `admin/` à 3 états (`locked → overview → teacherDetail`).
- `api/admin.ts` : code secret comparé en `timingSafeEqual` → cookie 12 h ; `overview` = `HGETALL admin:snapshots` ; détail = pipeline 2 GET + colonne retard recalculée côté client admin avec les **mêmes modules purs**.

## Phase 7 — PWA & push

- `vite-plugin-pwa` `injectManifest` ; `pwa/sw.ts` : precache + `NavigationRoute` avec `denylist [/^\/admin/, /^\/api\//]` + handlers `push`/`notificationclick`.
- `api/notify.ts` : POST subscribe/test (cookie) ; GET = **cron** (`Bearer CRON_SECRET`, `?dry=1` pour tester) → aujourd'hui férié/vacances/absence ? silence ; le prof enseigne-t-il aujourd'hui ? sinon silence ; retard ≥ seuil + anti-spam 2 j → web-push VAPID, purge des abonnements 404/410.

## Phase 8 — Mobile & finitions

- Remarque visible < 768 px (sous le contenu), FAB « + » safe-area, cibles ≥ 44 px, hover-reveal → toujours visible au tactile, bottom-sheets avec poignée + scroll du champ focusé.
- Barre de sélection contextuelle (voir composant), réordonnancement Monter/Descendre avec sélection préservée.

## Les moteurs JSON — colonne vertébrale des données

| Moteur | Fichier | Rôle |
|---|---|---|
| **Import / normalisation** | `utils/importPipeline.ts` (+ Web Worker `importWorkerHelper.ts`) | Tout JSON entrant (import manuel, sauvegarde, contenu prédéfini) passe par `prepareImportedLessons` : détection du format (tableau direct / `{lessonsData}` / `{data}`), normalisation des dates (fr → ISO), nettoyage des chaînes, réparation des conteneurs. Un rapport chiffré est retourné. Les gros fichiers sont traités **hors du thread principal**. |
| **Export classe** | `Editor.handleExportData` | `{classInfo, lessonsData}` → fichier `.json` téléchargé — réimportable tel quel. |
| **Export/restauration plateforme** | `Dashboard.handleExportPlatform` / `ImportPlatformModal` | `{config, classes:[{classInfo, lessonsData}]}` — sauvegarde intégrale, restauration + rechargement. |
| **Calendrier scolaire** | `public/vacances-jourferie.json` → `utils/calendar.ts` | Fériés + vacances, éditable sans rebuild (fetch no-cache côté client, import statique côté serveur). |
| **Contenus prédéfinis** | `public/contenus/manifest.json` + fichiers → `utils/predefinedContent.ts` | Programmes officiels par *niveau × matière* proposés sur cahier vide ; chargés via le MÊME pipeline d'import (normalisation garantie). |
| **Mémoire d'impression** | `utils/printMeta.ts` (`printMeta_v1_{classId}`) | Dates déjà imprimées par classe → la modale d'impression propose « nouveautés seulement ». |
| **Journal d'édition** | `utils/journal.ts` (`editJournal_v1_{classId}`) | Chaque opération (`operationType` du `useHistoryState`) horodatée, plafonnée à 60 → ligne « Dernière modification » + modale Historique. |

## Les 10 règles d'or retenues

1. Offline-first ; le cloud est une copie, jamais un préalable.
2. Modules de calcul **purs et partagés** client/serveur.
3. L'admin lit des **instantanés**, jamais les données brutes.
4. Toute date saisie passe par **la garde intelligente** ; alerter, ne jamais bloquer.
5. Une clé Redis **par classe** ; pipeline pour écrire, `HGETALL` pour lister.
6. Fonctions serverless **consolidées** (switch d'actions) — quota Hobby.
7. Fuseau **Africa/Casablanca** partout côté cron (serverless = UTC).
8. Tailwind : chaînes de classes **complètes** uniquement.
9. Les icônes de config sont des **composants**, pas des chaînes.
10. Chaque phase livrable seule ; `npm run lint && npm run build` verts avant de passer à la suivante.
